# Claude Code 会话管理器 — 操作记录

## 项目概述
开发一个 macOS Finder 风格的 Claude Code 会话管理器，支持按项目查看对话记录、在对应项目目录启动 Claude Code 继续工作。

- [x] 前端 Finder 风格 UI 正常渲染
- [x] 侧栏显示所有项目（按项目目录名）
- [x] 点击项目查看该项目的所有会话记录
- [x] 每个会话显示首条消息、时间、消息数、状态
- [x] 支持"继续最近"、"恢复会话"、"分叉会话"操作
- [x] 搜索功能跨项目搜索对话内容
- [x] 暗色模式自动适配（跟随系统设置）
- [x] 键盘快捷键（⌘K 搜索、⌘R 刷新）
- [x] 新建项目功能（选择任意文件夹开始工作）

## 数据源

| 数据源 | 路径 | 用途 |
|--------|------|------|
| 历史索引 | `~/.claude/history.jsonl` | 按 project/sessionId 分组，提供项目和会话索引 |
| 会话消息 | `~/.claude/projects/<encoded>/<sessionId>.jsonl` | 每个会话的完整消息记录 |
| 活跃会话 | `~/.claude/sessions/<pid>.json` | 当前运行中会话的状态信息 |
| 自定义项目 | `~/.claude/custom_projects.json` | 用户手动添加的项目文件夹（无历史记录的项目） |

## 技术架构

### 后端（session-manager.py）
- **框架**: Flask 3.x
- **数据读取**: 
  - `load_history()`: 解析 history.jsonl
  - `build_path_map()`: 建立编码路径 → 原始路径的反向映射
  - `get_session_status()`: 扫描 sessions 目录获取活跃状态
  - `get_first_message()`: 流式读取会话 JSONL 提取首条用户消息
- **API 端点**（10个）:
  - `GET /api/projects` — 项目列表（含历史项目和自定义项目）
  - `GET /api/projects/<encoded>/sessions` — 某项目的所有会话
  - `GET /api/projects/<encoded>/sessions/<id>/messages` — 会话消息详情（分页）
  - `POST /api/launch` — 启动 Claude Code（cmux 优先，Terminal 回退）
  - `GET /api/search?q=` — 跨项目搜索
  - `GET /api/stats` — 统计概览
  - `POST /api/select-folder` — 打开 macOS 原生文件夹选择器
  - `GET/POST/DELETE /api/custom-projects` — 管理用户自定义项目
  - `GET/POST /api/terminal-config` — 终端偏好管理（cmux / Terminal.app）
  - `POST /api/shutdown` — 停止 Flask 服务
- **启动方式**: 通过 AppleScript 调用 Terminal.app 执行 `claude -c` / `claude --resume <id>`

### 前端（HTML + CSS + JS）
- **布局**: CSS Grid 三区布局（标题栏 + 主内容区 + 状态栏）
- **主内容区**: 左侧 220px 侧栏 + 右侧内容区（Flexbox/Grid）
- **Finder 风格要素**:
  - 红黄绿窗口按钮（hover 显示关闭/最小化/最大化符号）
  - 毛玻璃效果（`backdrop-filter: blur(20px)`）
  - 侧栏选中蓝色高亮（`#0060C4` 浅色 / `#0A84FF` 暗色）
  - SF 系统字体栈（`-apple-system`）
  - 平滑过渡动画（150ms）
  - 会话卡片圆角 + 阴影 + hover 上浮
- **暗色模式**: 通过 `prefers-color-scheme: dark` 媒体查询自动切换
- **交互**:
  - 点击侧栏项目 → 加载会话列表
  - 点击会话卡片 → 右侧滑入消息详情面板
  - 搜索框实时搜索（300ms 防抖）
  - 键盘快捷键支持

### 8. cmux 终端集成（2026-05-04）

**需求**: 将会话管理器与 cmux 终端集成，点击"继续"/"新建"/"恢复"时在 cmux 中打开 Claude Code。

**cmux 限制**:
- cmux CLI (`new-workspace --command`) 仅接受从 cmux 内部启动的进程连接（Unix socket 访问控制）
- cmux 无 AppleScript 字典，只能通过 System Events 模拟按键（需辅助功能权限，macOS Sequoia 上受限）
- `open -a cmux <目录>` 可以打开工作区，但无法自动执行命令

**实现方案**（双模式）:
1. **cmux CLI 模式**（Flask 在 cmux 内运行时自动启用）：
   - 使用 `cmux new-workspace --cwd <项目> --command <命令>` 打开工作区并执行命令
   - 检测方式：`cmux ping` 返回码
2. **cmux 目录模式**（Flask 不在 cmux 内时的回退）：
   - 使用 `open -a cmux <项目目录>` 打开工作区
   - 同时将 Claude 命令复制到剪贴板（`pbcopy`）
   - 用户只需在 cmux 中按 `⌘V` + 回车

**后端修改 (session-manager.py)**:
- 新增 `CMUX_CLI`、`CMUX_APP` 常量
- 新增 `load_terminal_config()` / `save_terminal_config()` — 终端偏好持久化
- 新增 `get_effective_terminal()` — 返回当前终端选择
- 新增 `cmux_cli_works()` — 检测 cmux CLI 是否可用
- 修改 `/api/launch` — 支持 cmux 两种模式 + Terminal.app 回退
- 新增 `GET/POST /api/terminal-config` — 终端偏好管理

**前端修改 (app.js)**:
- `loadTerminalConfig()` — 加载终端状态
- `updateStatusBar()` — 显示当前终端模式（cmux / cmux(目录模式) / Terminal）

**使用方式**:
- **推荐**：在 cmux 中启动 Flask 服务（`python3 session-manager.py`），获得完整 cmux CLI 体验
- **桌面图标模式**：ccSession.app 启动 Flask → Web 界面操作 → cmux 打开目录 + 命令复制到剪贴板

### 9. cmux 自动进入会话改进（2026-05-04）

**需求**: 在会话管理器中点击“继续 / 新建 / 恢复”时，不再停留在“打开 cmux 后把命令复制到剪贴板，用户再手动粘贴回车”的半自动流程，而是要直接在 `cmux` 中执行对应 `claude` 命令。

**问题定位**:
- 原实现把 `cmux` 分成两种路径：
  - `cmux CLI` 可用：`cmux new-workspace --cwd ... --command ...`
  - `cmux CLI` 不可用：`open -a cmux <目录>` + `pbcopy`
- 本机实际行为是：即使 `cmux.app` 已安装并可被脚本控制，`cmux ping` 仍可能返回 `Broken pipe`，导致后端误判 CLI 不可用，直接退化成剪贴板方案。
- 检查 `cmux.app` 包内容后发现它实际带有 AppleScript 词典（`cmux.sdef`），支持：
  - `new window`
  - `new tab`
  - `activate window`
  - `input text`
  - `focus`
- 因此不需要再依赖“复制命令给用户手动粘贴”。

**后端修改 (`session-manager.py`)**:
- 新增 `run_osascript()`：
  - 统一封装 `osascript` 调用，支持通过 `argv` 安全传参，避免路径和命令字符串转义混乱。
- 新增 `launch_cmux_via_applescript(project_path, command)`：
  - 激活 `cmux`
  - 创建新的 `cmux` 窗口
  - 等待目标 terminal 就绪
  - 使用 `input text (shellCommand & linefeed)` 自动输入并回车执行命令
  - 尝试 `focus currentTerminal`，让新开的终端获得焦点
- 修改 `/api/launch` 的 `cmux` 回退逻辑：
  - 保留 `cmux CLI` 成功时的原生路径
  - 当 `cmux CLI` 不可用时，不再执行 `pbcopy`
  - 改为直接调用 `launch_cmux_via_applescript(...)`

**为什么 `linefeed` 很关键**:
- 实测 `input text "cd /tmp"` 只会“粘贴文本”，不会自动执行。
- 改成 `input text ("cd /tmp" & linefeed)` 后，命令会立刻执行，terminal 的工作目录会实际切换到 `/tmp`。
- 这一步是把“自动粘贴”提升为“自动执行”的关键。

**注意事项 / 已知现象**:
- `cmux` 的 AppleScript 接口可以稳定创建 terminal 并执行命令，但“最终哪个窗口成为 frontmost”受 `cmux` 自身窗口调度影响，不总能从脚本层 100% 强制为最前台窗口。
- 这不会影响命令自动执行本身；验证中已通过 terminal 数量和工作目录枚举确认终端确实被创建且命令已运行。
- 如果后续想继续压缩这个边界，优先方向应是：
  - 研究 `cmux` 可用 socket 的精确定位方式，恢复 `cmux CLI` 全能力控制；
  - 或继续挖掘 `cmux.sdef` 之外是否还有更强的窗口聚焦接口。

### 10. 恢复会话只打开 cmux、未真正进入 Claude 会话的修复（2026-05-04）

**现象**:
- 用户反馈：点击“恢复会话”按钮后，`cmux` 会被打开，但不会真正进入对应的 Claude Code 会话。

**排查过程**:

1. **先确认是不是 `cmux` 没有收到命令**
```bash
python3 - <<'PY'
... 调用 launch_cmux_via_applescript(..., "touch /tmp/cmux-launch-marker-1")
PY
```
- 结果：`/tmp/cmux-launch-marker-1` 被成功创建。
- 结论：`cmux` 的 AppleScript 注入链路本身是通的，不是“只打开了窗口完全没发命令”。

2. **再确认是不是 `claude` 命令本身失败**
```bash
python3 - <<'PY'
... 在 cmux 中执行:
claude --resume 1505765b-bd72-430b-8849-b2d1bbd59503 > /tmp/cmux-resume.txt 2>&1
PY
```
- 输出内容：
```text
No conversation found with session ID: 1505765b-bd72-430b-8849-b2d1bbd59503
```
- 这一步说明：命令确实执行了，但恢复参数不对。

3. **继续核对当前版本 Claude CLI 对 `--resume` 的真实要求**
```bash
claude --help
claude -p --resume "5b5dcebb-2ce3-40db-9874-af5f26d5f72e" "请只输出OK"
claude --resume "5b5dcebb-2ce3-40db-9874-af5f26d5f72e" </dev/null
```

**关键发现**:
- `claude -p --resume <session-id> "请只输出OK"` 可以成功，输出 `OK`
- 但仅执行：
```bash
claude --resume <session-id>
```
会报错：
```text
Error: No deferred tool marker found in the resumed session. Either the session was not deferred, the marker is stale (tool already ran), or it exceeds the tail-scan window. Provide a prompt to continue the conversation.
```

**最终原因**:
- 当前版本 Claude CLI 恢复普通历史会话时，不能只传裸 `session-id`。
- 除了 `--resume <session-id>` 之外，还必须再附带一个继续提示词，才能真正进入会话。
- 原会话管理器构造的是：
```bash
claude --resume <session-id>
```
- 所以点击恢复后看起来像“只打开了 cmux”，实际是 Claude 立即报错退出了。

**代码修复 (`session-manager.py`)**:
- 新增常量：
```python
DEFAULT_RESUME_PROMPT = "继续这个会话。"
```
- 修改 `resume` 分支命令生成逻辑：
```python
claude --resume <session-id> '继续这个会话。'
```
- 对 `fork-session` 同样生效。

### 12. 改为本地应用窗口运行，不再依赖浏览器（2026-05-14）

### 6. 桌面快捷应用开发（2026-05-03）

**需求**: 将会话管理器包装为 macOS .app 应用，可放在桌面或程序坞，点击即用。

**创建命令**:
```bash
osacompile -o ~/Desktop/ccSession.app -e '
set portCheck to do shell script "lsof -ti:5199 2>/dev/null || echo \"\""
if portCheck is not "" then
    do shell script "open http://127.0.0.1:5199"
else
    do shell script "cd \"/path/to/project\" && python3 session-manager.py > /dev/null 2>&1 &"
end if
'
```

**ccSession.app 结构**:
```
ccSession.app/
└── Contents/
    ├── Info.plist
    ├── PkgInfo
    ├── _CodeSignature/CodeResources
    ├── MacOS/applet              # 编译后的原生可执行文件
    └── Resources/
        ├── applet.icns           # 默认图标
        ├── applet.rsrc
        └── Scripts/main.scpt     # AppleScript 源码
```

**工作流程**:
1. 用户双击 → 执行 AppleScript

## 后端修改详情

### 1. GET /api/active-sessions

业务流程：
1. 扫描 `~/.claude/sessions/` 目录下所有 `*.json` 状态文件
2. 从 history.jsonl 构建 session_info 查找表（sessionId → projectPath/projectName/firstDisplay）
3. 对每个活跃会话：解析 sessionId、pid、cwd、status
4. 尝试从 JSONL 文件读取标题（调用现有 parse_session_meta）
5. 返回活跃会话列表，按标题排序

响应格式：
```json
[{
    "sessionId": "uuid",
    "pid": 12345,
    "cwd": "/path/to/project",
    "projectPath": "/path/to/project",
    "projectName": "项目名",
    "encoded": "-path-encoded",
    "status": "idle|busy|waiting",
    "title": "首条用户消息..."
}]
```

### 2. POST /api/sessions/<session_id>/exit

业务流程：
1. 扫描 sessions 目录找到匹配的状态文件
2. 读取 PID
3. `os.killpg(os.getpgid(pid), signal.SIGTERM)` 终止整个进程组
4. `ProcessLookupError` 静默处理（进程已不存在）
5. 删除状态文件
6. 返回成功响应

错误处理：
- 会话不存在 → 404
- 无 PID → 400
